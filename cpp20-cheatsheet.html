Modern C++20 — Cheat Sheet & Interview Reference

A single-file reference covering language fundamentals, the STL, modern idioms, and C++20 additions. Designed for someone with prior C++ experience refreshing before a technical interview. Hover over any code block to copy.

Why & when C++

C++ is a multi-paradigm, statically typed, compiled systems language with zero-overhead abstraction as its guiding principle: you should not pay (in runtime cost or memory) for features you don't use, and what you do use should be as efficient as hand-written C. It gives you direct control of memory layout, lifetime, and CPU instructions while still offering high-level abstractions (templates, RAII, the STL).

What it does best

  • Latency-critical work. Game engines (Unreal), trading systems, audio/DSP, robotics, real-time control. You can reason about cache behavior, allocation, and branch cost.
  • Systems software. Databases (ClickHouse, MySQL, MongoDB), browsers (Chromium, WebKit), compilers (LLVM, GCC), operating system components.
  • Numerical / HPC. Scientific computing, ML kernels (PyTorch C++ core, TensorFlow), CAD/simulation, financial pricing libraries.
  • Embedded with abstraction. Where C is too low-level but you still want deterministic behavior and no garbage collector.
  • Cross-language plumbing. Native libraries that everything else binds to (Python C extensions, Node addons).

Where it falls short

  • Iteration speed. Long build times, slow feedback loops vs Python/Go/TypeScript. Templates and headers compound this.
  • Easy footguns. Undefined behavior, dangling references, data races, lifetime bugs. The language trusts you completely.
  • Dependency management. No standard package manager; CMake/Conan/vcpkg are an ecosystem to learn separately.
  • Surface area. The language is huge. Every feature interacts with every other; corner cases abound.
  • Web/scripting/glue. Almost never the right choice. Reach for Python, TypeScript, Go.
Rule of thumb for choosing C++: if you would lose money or break determinism by using a garbage-collected language, C++ is a contender. Otherwise, a higher-level language is usually a better default.

Mental model

Three ideas account for most of "thinking in modern C++":

  1. Values, not references, by default. Variables are objects, not handles to objects. Assignment copies. Passing copies. Move semantics let you transfer ownership cheaply when you want it.
  2. RAII: lifetime is the unit of resource management. Constructors acquire, destructors release. Stack unwinding cleans up on exceptions. Smart pointers, locks, file handles, sockets all follow this.
  3. Templates are compile-time code generation. The compiler stamps out a new function/class for each set of template arguments. Concepts (C++20) put constraints on those arguments.

Variables, auto, const, constexpr

// Initialization forms
int a = 5;            // copy init
int b(5);             // direct init
int c{5};             // brace (uniform) init — preferred; rejects narrowing
int d = {5};          // copy-list init
// int e{3.14};       // ERROR: narrowing
int f{};              // value init → 0

// auto deduces from initializer
auto v = std::vector<int>{1, 2, 3};
const auto& cref = v;  // const reference to v

// Structured bindings (C++17, often used in C++20)
std::pair<int, std::string> p{1, "x"};
auto [id, name] = p;

std::map<std::string, int> m{{"a", 1}, {"b", 2}};
for (const auto& [k, val] : m) { /* ... */ }

// const — "I won't change this object"
const int  N = 100;
const int* p1;   // pointer to const int (data is const)
int* const p2 = &a; // const pointer to int (pointer is const)
const int* const p3 = &a;

// constexpr — "evaluable at compile time"
constexpr int square(int x) { return x * x; }
constexpr int s = square(5);   // computed at compile time

// consteval (C++20) — MUST run at compile time
consteval int cube(int x) { return x * x * x; }
constexpr int k = cube(3);     // ok
// int n = cube(rand());       // ERROR

// constinit (C++20) — runtime init forbidden, but value is mutable
constinit int counter = 0;     // static storage, no init-order fiasco

auto, decltype, decltype(auto)

int  n = 5;
int& r = n;
auto a1 = r;          // int  (auto drops reference / top-level const)
auto& a2 = r;         // int&
decltype(r)  d1 = n;  // int&  (preserves reference)
decltype(n)  d2 = n;  // int

// decltype(auto) preserves reference-ness from the return expression
decltype(auto) get(std::vector<int>& v) { return v[0]; }  // returns int&

// Modern guidance:
// • Pass by value if cheap to copy (int, std::string_view, std::span).
// • Pass by const ref for non-trivial inputs you won't modify.
// • Pass by non-const ref for "in-out" parameters.
// • Pass by rvalue ref (T&&) when you specifically want to take ownership.
// • Pass raw T* only for optional / nullable non-owning.
// • Pass std::unique_ptr<T> / shared_ptr to transfer ownership.

Control flow

if / else — including init-statement and if constexpr

if (int n = compute(); n > 0) {       // init-statement (C++17)
    use(n);
} else if (n == 0) {
    // ...
} else {
    // ...
}

// Compile-time branch — non-taken branch isn't instantiated
template <typename T>
auto to_string(const T& x) {
    if constexpr (std::is_integral_v<T>)
        return std::to_string(x);
    else if constexpr (std::is_same_v<T, std::string>)
        return x;
    else
        return std::string{"?"};
}

switch

switch (status) {
    case 200:
    case 204:
        ok();
        break;
    case 404:
        not_found();

    case 500:
        log_error();

    default:
        unknown();
}

Loops

for (int i = 0; i < 10; ++i) { /* classic */ }

while (cond) { /* ... */ }
do { /* ... */ } while (cond);

// Range-based for — works on anything with begin()/end()
std::vector<int> v{1,2,3};
for (int x : v)        { /* copy */ }

// Reverse with views (C++20)
import std;  // or: #include <ranges>
for (int x : v | std::views::reverse) { /* ... */ }

Inline if — the ternary operator ? :

An inline if (ternary / conditional operator) evaluates to one of two values in a single expression: condition ? value_if_true : value_if_false. Use it for simple value selection; prefer a regular if/else for anything with side effects or multiple statements.

int x = 10;

// Inline if — basic form
int abs_x = (x < 0) ? -x : x;          // -10 if negative, 10 otherwise

// Assign to a variable
std::string label = (x % 2 == 0) ? "even" : "odd";

// Inline in a function call argument
std::cout << "Value: " << (x > 0 ? "positive" : "non-positive") << '\n';

// Return directly from a function
int clamp(int v, int lo, int hi) {
    return (v < lo) ? lo : (v > hi) ? hi : v;  // nested ternary — keep shallow
}

// const / constexpr — ternary works at compile time
constexpr int limit = (sizeof(void*) == 8) ? 1024 : 512;  // 64 vs 32-bit
Nested ternaries compile but hurt readability fast — more than one level deep, switch to if/else if. Also note: both branches must be of compatible types; mismatched types cause an implicit conversion that can surprise you (e.g. true ? 0 : 0.5 yields double).

Functions & lambdas

// Trailing return type — useful with templates
auto add(int a, int b) -> int { return a + b; }

// Default arguments
void greet(std::string_view name = "world");

// Overloading
int  area(int side);
double area(double r);

// Pass-by-value-then-move idiom (one function handles lvalues & rvalues)
class Person {
    std::string name_;
public:
    explicit Person(std::string name) : name_(std::move(name)) {}
};

// [[nodiscard]], [[deprecated]], [[noreturn]]
[[nodiscard]] int parse(std::string_view s);
[[noreturn]] void die(const char* msg);

// Lambdas
auto sq = [](int x) { return x * x; };

// Captures
int z = 5;
auto by_val   = [z](int x)        { return x + z; };
auto by_ref   = [&z](int x)       { return x + z; };
auto by_move  = [s = std::string{"hi"}](int){ return s.size(); };
auto by_all_v = [=]() mutable { /* copies of all used names */ };
auto by_all_r = [&]() { /* refs to all used names — danger of dangling */ };

// Generic lambda (C++14) / template lambda (C++20)
auto gen   = [](auto x, auto y) { return x + y; };
auto tmpl  = []<typename T>(std::vector<T>& v) { return v.size(); };

// std::function — type-erased callable (heap alloc, virtual call)
std::function<int(int)> f = sq;
Lambda over std::function: pass lambdas as templates (or via auto parameters) to keep them inlined and allocation-free. Reach for std::function only when you need type erasure (e.g., storing heterogeneous callbacks).

Errors & exceptions

#include <stdexcept>

int parse(std::string_view s) {
    if (s.empty()) throw std::invalid_argument("empty");
    // ...
    return 0;
}

try {
    parse("");
} catch (const std::invalid_argument& e) {
    std::cerr << "bad input: " << e.what() << '\n';
} catch (const std::exception& e) {
    std::cerr << e.what() << '\n';
} catch (...) {
    std::cerr << "unknown error\n";
}

// noexcept — promise not to throw; enables optimizations and move-vs-copy choices
void close() noexcept;
auto sum(const std::vector<int>& v) noexcept -> int;

// Alternatives to exceptions (common in performance-sensitive code):
std::optional<int> try_parse(std::string_view);
std::expected<int, ParseError> parse2(std::string_view);  // C++23

// Error codes (C++17 onward)
#include <system_error>
std::error_code ec;
auto f = std::filesystem::file_size("x", ec);
Destructors must not throw. If a destructor throws during stack unwinding caused by another exception, std::terminate is called. Mark them noexcept (the default) and swallow / log errors instead.

All substrings of a string

std::string s = "abc";

// Every substring — outer loop picks start, inner picks length
// substr(pos, len): pos = starting index, len = number of characters
for (std::size_t i = 0; i < s.size(); ++i) {
    for (std::size_t len = 1; len <= s.size() - i; ++len) {
        std::cout << s.substr(i, len) << '\n';
        // i=0,len=1 → "a"
        // i=0,len=2 → "ab"
        // i=0,len=3 → "abc"
        // i=1,len=1 → "b"
        // i=1,len=2 → "bc"
        // i=2,len=1 → "c"
    }
}

// Collect into a vector
std::vector<std::string> subs;
for (std::size_t i = 0; i < s.size(); ++i)
    for (std::size_t len = 1; len <= s.size() - i; ++len)
        subs.push_back(s.substr(i, len));

// Zero-copy version using string_view (avoids allocating each substring)
std::vector<std::string_view> views;
for (std::size_t i = 0; i < s.size(); ++i)
    for (std::size_t len = 1; len <= s.size() - i; ++len)
        views.emplace_back(s.data() + i, len);  // s must outlive views

The three main functions

#include <regex>
#include <string>

std::string s = "Order #1042 placed on 2024-05-10";
std::regex  date_re(R"(\d{4}-\d{2}-\d{2})");   // raw string avoids double-escaping

// ── regex_match ───────────────────────────────────────────
// Returns true only if the ENTIRE string matches the pattern.
std::regex word_re(R"(\w+)");
bool whole = std::regex_match("hello", word_re);   // true
bool part  = std::regex_match(s, date_re);          // false (more than just a date)

// ── regex_search ──────────────────────────────────────────
// Returns true if the pattern appears ANYWHERE in the string.
std::smatch m;
if (std::regex_search(s, m, date_re)) {
    std::cout << "Found: " << m[0] << '\n';    // "2024-05-10"
    std::cout << "Prefix: " << m.prefix() << '\n'; // text before the match
    std::cout << "Suffix: " << m.suffix() << '\n'; // text after the match
}

// ── regex_replace ─────────────────────────────────────────
// Replace every match; $0 = whole match, $1 = first group, etc.
std::regex num_re(R"(\d+)");
std::string result = std::regex_replace(s, num_re, "[NUM]");
// "Order #[NUM] placed on [NUM]-[NUM]-[NUM]"

Capture groups

// Groups are accessed via m[1], m[2], … (m[0] is always the full match)
std::regex log_re(R"((\d{4}-\d{2}-\d{2}) (\w+): (.+))");
std::string log = "2024-05-10 ERROR: disk full";
std::smatch gm;
if (std::regex_search(log, gm, log_re)) {
    std::cout << "Date:    " << gm[1] << '\n';  // "2024-05-10"
    std::cout << "Level:   " << gm[2] << '\n';  // "ERROR"
    std::cout << "Message: " << gm[3] << '\n';  // "disk full"
}

Iterating over all matches

std::string text = "foo 123 bar 456 baz 789";
std::regex  nums(R"(\d+)");

// sregex_iterator walks every non-overlapping match
for (auto it  = std::sregex_iterator(text.begin(), text.end(), nums),
          end = std::sregex_iterator{};
     it != end; ++it) {
    std::cout << (*it)[0] << '\n';   // 123, 456, 789
}

// Collect all matches into a vector
std::vector<std::string> matches;
std::transform(
    std::sregex_iterator(text.begin(), text.end(), nums),
    std::sregex_iterator{},
    std::back_inserter(matches),
    [](const std::smatch& m){ return m[0].str(); }
);

std::vector — the workhorse

#include <vector>
std::vector<int> v;             // empty
std::vector<int> v2(10);        // 10 zeros
std::vector<int> v3(10, 7);     // 10 sevens
std::vector<int> v4{1, 2, 3};   // brace-init list

v.reserve(1000);                // pre-allocate capacity (no size change)
v.resize(50);                   // change size (default-construct extras)

v.push_back(42);
v.emplace_back(42);             // constructs in place — preferred for non-trivial T

v.pop_back();
v.front(); v.back();
v[i]; v.at(i);                  // at() throws std::out_of_range

v.size(); v.capacity(); v.empty();
v.clear();                      // size=0, capacity unchanged
v.shrink_to_fit();              // hint to release capacity

// Insert / erase
v.insert(v.begin() + 2, 99);
v.erase(v.begin());             // O(n)
v.erase(std::remove(v.begin(), v.end(), 7), v.end());  // erase-remove

// Or C++20 free functions:
std::erase(v, 7);
std::erase_if(v, [](int x){ return x % 2 == 0; });

// Move semantics matter here — reserve to avoid reallocs invalidating refs
auto& first = v.front();
v.push_back(1);                 // may invalidate `first`!

Algorithm patterns

Recurring problem-solving techniques used in competitive programming and technical interviews. Each pattern is a structural template; recognising which one applies is half the solution.

Sliding window

Maintain a contiguous sub-array/sub-string that satisfies a condition. Expand the right edge; shrink the left edge when the condition is violated. O(n) instead of O(n²) brute force.

// Fixed window: max sum of k consecutive elements
int maxSumFixed(const std::vector<int>& v, int k) {
    int window = 0;
    for (int i = 0; i < k; ++i) window += v[i];
    int best = window;
    for (int i = k; i < (int)v.size(); ++i) {
        window += v[i] - v[i - k];
        best = std::max(best, window);
    }
    return best;
}

// Variable window: longest sub-array with sum <= target
int longestSubarrayLE(const std::vector<int>& v, int target) {
    int left = 0, sum = 0, best = 0;
    for (int right = 0; right < (int)v.size(); ++right) {
        sum += v[right];
        while (sum > target) sum -= v[left++];  // shrink left
        best = std::max(best, right - left + 1);
    }
    return best;
}

// Variable window on string: longest substring with at most k distinct chars
int longestKDistinct(const std::string& s, int k) {
    std::unordered_map<char, int> freq;
    int left = 0, best = 0;
    for (int right = 0; right < (int)s.size(); ++right) {
        ++freq[s[right]];
        while ((int)freq.size() > k) {
            if (--freq[s[left]] == 0) freq.erase(s[left]);
            ++left;
        }
        best = std::max(best, right - left + 1);
    }
    return best;
}

Two pointers

Two indices moving through the same (or two) arrays. Common forms: converging (left + right from ends), parallel (both from left), and fast/slow (cycle detection).

// Converging: two-sum on sorted array
std::pair<int,int> twoSumSorted(const std::vector<int>& v, int target) {
    int l = 0, r = (int)v.size() - 1;
    while (l < r) {
        int s = v[l] + v[r];
        if (s == target) return {l, r};
        else if (s < target) ++l;
        else                  --r;
    }
    return {-1, -1};
}

// Parallel: merge two sorted arrays
std::vector<int> mergeSorted(const std::vector<int>& a, const std::vector<int>& b) {
    std::vector<int> out;
    out.reserve(a.size() + b.size());
    int i = 0, j = 0;
    while (i < (int)a.size() && j < (int)b.size())
        out.push_back(a[i] <= b[j] ? a[i++] : b[j++]);
    out.insert(out.end(), a.begin() + i, a.end());
    out.insert(out.end(), b.begin() + j, b.end());
    return out;
}

// Fast / slow (Floyd's cycle detection)
bool hasCycle(ListNode* head) {
    auto* slow = head;
    auto* fast = head;
    while (fast && fast->next) {
        slow = slow->next;
        fast = fast->next->next;
        if (slow == fast) return true;
    }
    return false;
}

Prefix sum

Precompute cumulative sums so any sub-array sum is O(1). Also used for 2-D grids and XOR prefix arrays.

// Build prefix array; query range sum [l, r] inclusive
std::vector<int> buildPrefix(const std::vector<int>& v) {
    std::vector<int> pre(v.size() + 1, 0);
    for (int i = 0; i < (int)v.size(); ++i)
        pre[i + 1] = pre[i] + v[i];
    return pre;          // pre[r+1] - pre[l] = sum(v[l..r])
}

// Count sub-arrays with sum == k  (hash map of prefix sums)
int subarraySum(const std::vector<int>& v, int k) {
    std::unordered_map<int, int> seen{{0, 1}};
    int count = 0, prefix = 0;
    for (int x : v) {
        prefix += x;
        count += seen[prefix - k];   // how many times (prefix - k) appeared
        ++seen[prefix];
    }
    return count;
}

// 2-D prefix sum for rectangle queries
// pre[i][j] = sum of rectangle (0,0)..(i-1,j-1)
std::vector<std::vector<int>> prefix2D(const std::vector<std::vector<int>>& g) {
    int R = g.size(), C = g[0].size();
    std::vector<std::vector<int>> p(R+1, std::vector<int>(C+1, 0));
    for (int r = 1; r <= R; ++r)
        for (int c = 1; c <= C; ++c)
            p[r][c] = g[r-1][c-1] + p[r-1][c] + p[r][c-1] - p[r-1][c-1];
    return p;
    // query (r1,c1)..(r2,c2):  p[r2+1][c2+1] - p[r1][c2+1] - p[r2+1][c1] + p[r1][c1]
}

HashMap — frequency & complement lookup

Trade O(n) space for O(1) lookup. Common uses: character frequency, "have I seen the complement of this value?", grouping anagrams.

// Two-sum (unsorted): find indices where v[i] + v[j] == target
std::pair<int,int> twoSum(const std::vector<int>& v, int target) {
    std::unordered_map<int,int> seen;   // value -> index
    for (int i = 0; i < (int)v.size(); ++i) {
        int need = target - v[i];
        if (auto it = seen.find(need); it != seen.end())
            return {it->second, i};
        seen[v[i]] = i;
    }
    return {-1, -1};
}

// Character frequency (anagram check)
bool isAnagram(const std::string& a, const std::string& b) {
    if (a.size() != b.size()) return false;
    std::unordered_map<char,int> freq;
    for (char c : a) ++freq[c];
    for (char c : b) if (--freq[c] < 0) return false;
    return true;
}

// Group anagrams: words with same sorted form share a key
std::vector<std::vector<std::string>> groupAnagrams(std::vector<std::string>& words) {
    std::unordered_map<std::string, std::vector<std::string>> groups;
    for (auto& w : words) {
        auto key = w;
        std::sort(key.begin(), key.end());
        groups[key].push_back(w);
    }
    std::vector<std::vector<std::string>> result;
    for (auto& [_, g] : groups) result.push_back(std::move(g));
    return result;
}

BFS (Breadth-First Search)

Explores level by level using a queue. Guarantees shortest path in an unweighted graph. Use for: shortest path, level-order tree traversal, multi-source flood fill.

#include <queue>

// Shortest path in unweighted graph (adjacency list)
int bfsDistance(const std::vector<std::vector<int>>& adj, int src, int dst) {
    int n = adj.size();
    std::vector<int> dist(n, -1);
    std::queue<int> q;
    dist[src] = 0;
    q.push(src);
    while (!q.empty()) {
        int u = q.front(); q.pop();
        if (u == dst) return dist[u];
        for (int nb : adj[u]) {
            if (dist[nb] == -1) {
                dist[nb] = dist[u] + 1;
                q.push(nb);
            }
        }
    }
    return -1;  // unreachable
}

// BFS on a 2-D grid (4-directional)
int bfsGrid(std::vector<std::vector<int>>& grid, int sr, int sc, int er, int ec) {
    static const int dr[]{-1,1,0,0}, dc[]{0,0,-1,1};
    int R = grid.size(), C = grid[0].size();
    std::queue<std::pair<int,int>> q;
    grid[sr][sc] = -1;   // mark visited in-place (or use a visited array)
    q.push({sr, sc});
    int steps = 0;
    while (!q.empty()) {
        for (int sz = q.size(); sz--; ) {
            auto [r, c] = q.front(); q.pop();
            if (r == er && c == ec) return steps;
            for (int d = 0; d < 4; ++d) {
                int nr = r + dr[d], nc = c + dc[d];
                if (nr>=0 && nr<R && nc>=0 && nc<C && grid[nr][nc] == 0) {
                    grid[nr][nc] = -1;
                    q.push({nr, nc});
                }
            }
        }
        ++steps;
    }
    return -1;
}

DFS (Depth-First Search)

Explores as deep as possible before backtracking. Use for: cycle detection, topological sort, connected components, permutations/combinations, tree paths.

// Recursive DFS — connected components
void dfs(int u, const std::vector<std::vector<int>>& adj, std::vector<bool>& visited) {
    visited[u] = true;
    for (int nb : adj[u])
        if (!visited[nb]) dfs(nb, adj, visited);
}

int countComponents(int n, const std::vector<std::vector<int>>& adj) {
    std::vector<bool> visited(n, false);
    int count = 0;
    for (int i = 0; i < n; ++i)
        if (!visited[i]) { dfs(i, adj, visited); ++count; }
    return count;
}

// Iterative DFS (avoids stack overflow on deep graphs)
void dfsIterative(int src, const std::vector<std::vector<int>>& adj) {
    std::vector<bool> visited(adj.size(), false);
    std::stack<int> st;
    st.push(src);
    while (!st.empty()) {
        int u = st.top(); st.pop();
        if (visited[u]) continue;
        visited[u] = true;
        for (int nb : adj[u]) if (!visited[nb]) st.push(nb);
    }
}

// Backtracking DFS: generate all permutations
void permute(std::vector<int>& nums, int start, std::vector<std::vector<int>>& result) {
    if (start == (int)nums.size()) { result.push_back(nums); return; }
    for (int i = start; i < (int)nums.size(); ++i) {
        std::swap(nums[start], nums[i]);
        permute(nums, start + 1, result);
        std::swap(nums[start], nums[i]);  // backtrack
    }
}

Top K elements

Use a heap (priority queue) to maintain the K best elements seen so far in O(n log k) instead of sorting in O(n log n).

#include <queue>

// Top K largest — min-heap of size k
std::vector<int> topKLargest(const std::vector<int>& v, int k) {
    std::priority_queue<int, std::vector<int>, std::greater<int>> minHeap;
    for (int x : v) {
        minHeap.push(x);
        if ((int)minHeap.size() > k) minHeap.pop();  // evict smallest
    }
    std::vector<int> result;
    while (!minHeap.empty()) { result.push_back(minHeap.top()); minHeap.pop(); }
    return result;  // ascending order — reverse if needed
}

// Kth largest element (same idea, just return top)
int kthLargest(const std::vector<int>& v, int k) {
    std::priority_queue<int, std::vector<int>, std::greater<int>> minHeap;
    for (int x : v) {
        minHeap.push(x);
        if ((int)minHeap.size() > k) minHeap.pop();
    }
    return minHeap.top();
}

// K most frequent elements — pair (freq, value) in max-heap
std::vector<int> topKFrequent(const std::vector<int>& v, int k) {
    std::unordered_map<int,int> freq;
    for (int x : v) ++freq[x];

    std::priority_queue<std::pair<int,int>> maxHeap;  // (freq, val)
    for (auto& [val, cnt] : freq) maxHeap.push({cnt, val});

    std::vector<int> result;
    for (int i = 0; i < k; ++i) { result.push_back(maxHeap.top().second); maxHeap.pop(); }
    return result;
}

// Merge K sorted lists — min-heap of (value, list-index, element-index)
using T3 = std::tuple<int,int,int>;
std::vector<int> mergeKSorted(const std::vector<std::vector<int>>& lists) {
    std::priority_queue<T3, std::vector<T3>, std::greater<T3>> pq;
    for (int i = 0; i < (int)lists.size(); ++i)
        if (!lists[i].empty()) pq.push({lists[i][0], i, 0});
    std::vector<int> out;
    while (!pq.empty()) {
        auto [val, li, ei] = pq.top(); pq.pop();
        out.push_back(val);
        if (ei + 1 < (int)lists[li].size())
            pq.push({lists[li][ei+1], li, ei+1});
    }
    return out;
}
Heap direction cheat sheet: top-K largest → min-heap of size K (pop the smallest intruder). Top-K smallest → max-heap of size K. Always process next minimum (Dijkstra, merge) → min-heap (std::greater).

Brian Kernighan's trick — n & (n - 1)

Clears the lowest set bit of n in one operation. Useful for counting set bits and checking powers of two.

// Count set bits (Hamming weight) — O(set bits), not O(bit width)
int countBits(int n) {
    int count = 0;
    while (n) { n &= (n - 1); ++count; }   // each iteration drops one set bit
    return count;
}

// Check if n is a power of two (exactly one bit set)
bool isPowerOfTwo(int n) {
    return n > 0 && (n & (n - 1)) == 0;
}

// Check if n is a power of two, or zero (bit cleared → 0 means ≤1 set bit)
bool atMostOneBit(unsigned n) { return (n & (n - 1)) == 0; }

Other common bit tricks

// Isolate the lowest set bit  (e.g. 0b10110 → 0b00010)
int lowest = n & (-n);         // -n == ~n + 1  (two's complement negation)

// Clear the lowest set bit    (Brian Kernighan, inline)
n &= (n - 1);

// XOR swap (no temp variable — avoid in practice; compilers do it better)
a ^= b; b ^= a; a ^= b;

// Find the single non-duplicate in an array where everything else appears twice
int findUnique(const std::vector<int>& v) {
    int x = 0;
    for (int n : v) x ^= n;   // pairs cancel out; unique survives
    return x;
}

// Check if two integers have opposite signs
bool oppositeSigns(int a, int b) { return (a ^ b) < 0; }

// Round up to next power of two (32-bit)
uint32_t nextPow2(uint32_t n) {
    --n;
    n |= n >> 1; n |= n >> 2; n |= n >> 4;
    n |= n >> 8; n |= n >> 16;
    return ++n;
}

Counting bits — __builtin_popcount and std::popcount

__builtin_* are GCC/Clang compiler intrinsics that map to a single CPU instruction where available. C++20 standardises the same operations in <bit>, which is portable and preferred in modern code.

Rule of 0 / 3 / 5 — copy & move

Six "special member functions" the compiler may generate:

  1. Default constructor — T()
  2. Destructor — ~T()
  3. Copy constructor — T(const T&)
  4. Copy assignment — T& operator=(const T&)
  5. Move constructor — T(T&&) noexcept
  6. Move assignment — T& operator=(T&&) noexcept
Rule of Zero: if all your members are RAII types (smart pointers, vectors, strings), define none of the special members — the defaults are correct.
Rule of Five: if you must define one of {dtor, copy, move}, you almost certainly need all five. 
class Buffer {
    std::size_t n_ = 0;
    int* data_ = nullptr;
public:
    Buffer() = default;
    explicit Buffer(std::size_t n) : n_(n), data_(new int[n]{}) {}

    ~Buffer() { delete[] data_; }

    Buffer(const Buffer& o) : n_(o.n_), data_(new int[o.n_]) {
        std::copy_n(o.data_, n_, data_);
    }
    Buffer& operator=(const Buffer& o) {
        if (this == &o) return *this;
        Buffer tmp(o);           // copy-and-swap idiom
        std::swap(n_, tmp.n_);
        std::swap(data_, tmp.data_);
        return *this;
    }

    Buffer(Buffer&& o) noexcept
        : n_(std::exchange(o.n_, 0)),
          data_(std::exchange(o.data_, nullptr)) {}
    Buffer& operator=(Buffer&& o) noexcept {
        std::swap(n_, o.n_);
        std::swap(data_, o.data_);
        return *this;
    }
};

// Or just delete them
class NonCopyable {
public:
    NonCopyable(const NonCopyable&) = delete;
    NonCopyable& operator=(const NonCopyable&) = delete;

Move semantics in two minutes

  • lvalue: has a name, persists. rvalue: temporary, about to die.
  • T&& binds to rvalues. Used for move constructors / assignment.
  • std::move(x) is just a static_cast<T&&>(x) — it doesn't move anything; it tells overload resolution "treat x as a temporary".
  • After std::move(x), x is in a valid but unspecified state. You can assign to it or destroy it; reading its value is a bug.
  • Move operations should be noexcept — std::vector only uses move during reallocation if it's noexcept; otherwise it copies.
std::vector<std::string> v;
std::string s = "hi";
v.push_back(s);               // copy
v.push_back(std::move(s));    // move — s now valid-but-unspecified
v.emplace_back("new");        // construct in place

// Perfect forwarding — preserve value category through a template
template <typename T>
void wrapper(T&& arg) {           // "forwarding reference"
    callee(std::forward<T>(arg));  // forwards as lvalue or rvalue as bound
}

Dynamic memory: new vs malloc / calloc

C++ offers two families of dynamic allocation. The C++ family (new / delete) is type-aware and integrates with constructors and exceptions. The C family (malloc / calloc / realloc / free) works in raw bytes with no type knowledge. Never mix the two — memory from malloc must be freed with free; memory from new must be released with delete.

#include <cstdlib>

// Overflow risk — large count * sizeof can wrap to small number

// ── calloc ────────────────────────────────────────────────
int* c = static_cast<int*>(calloc(10, sizeof(int)));  // 10 ints, all zero

// ── realloc ───────────────────────────────────────────────
// Only safe for trivially copyable types (no constructors/destructors)

// If realloc succeeds, c is invalid — use grown from here on

Smart pointers & RAII (Resource Acquisition Is Initialization)

The single most important modern C++ habit: never use raw new/delete. Use smart pointers and standard containers, which acquire in their constructor and release in their destructor.

#include <memory>

// unique_ptr — exclusive ownership, zero overhead, move-only
auto p = std::make_unique<Widget>(arg1, arg2);
p->method();
auto p2 = std::move(p);                  // transfer ownership
// p is now nullptr

std::unique_ptr<int[]> arr = std::make_unique<int[]>(100);  // dynamic array

// shared_ptr — shared ownership, reference-counted (atomic, has overhead)

auto sp2 = sp;                           // refcount: 2

// weak_ptr — non-owning observer of a shared_ptr; breaks ref cycles
std::weak_ptr<Widget> wp = sp;
if (auto strong = wp.lock()) { use(*strong); }

// Custom deleter
auto file = std::unique_ptr<FILE, decltype(&std::fclose)>(
    std::fopen("x", "r"), &std::fclose);

// Why prefer make_unique / make_shared:
//   - exception safety: f(new T, new U) can leak if one new throws
//   - make_shared: single allocation, better cache locality
//   - no naked `new`
Never: shared_ptr<T> sp(raw_ptr); shared_ptr<T> sp2(raw_ptr); — two control blocks, double-free. If you must wrap a raw pointer, do it exactly once.

Coroutines C++20

Coroutines are functions that can suspend and resume. C++20 ships the machinery (co_await, co_yield, co_return) but very little library — you typically use a coroutine library (cppcoro, asio, libunifex) or write your own promise types.

#include <coroutine>

// A bare-bones generator (C++23 has std::generator)
template <typename T>
struct Generator {
    struct promise_type {
        T current;
        Generator get_return_object() { return Generator{handle::from_promise(*this)}; }
        std::suspend_always initial_suspend() noexcept { return {}; }
        std::suspend_always final_suspend()   noexcept { return {}; }
        std::suspend_always yield_value(T v) { current = std::move(v); return {}; }
        void return_void() {}
        void unhandled_exception() { std::terminate(); }
    };
    using handle = std::coroutine_handle<promise_type>;
    handle h;
    explicit Generator(handle h) : h(h) {}
    ~Generator() { if (h) h.destroy(); }
    bool next() { h.resume(); return !h.done(); }
    T value() const { return h.promise().current; }
};

Generator<int> ints(int from, int to) {
    for (int i = from; i < to; ++i) co_yield i;
}

// Usage
for (auto g = ints(0, 5); g.next(); ) {
    std::cout << g.value() << '\n';
}

In practice, treat C++20 coroutines as a protocol the language supports; the actual abstractions (tasks, generators, async I/O) come from a library.

    std::unique_lock     ul(m);        // can unlock/re-lock, transfer to cv
    std::scoped_lock     sl(m, rm);    // multiple mutexes, deadlock-free
    std::shared_lock     sh(rwm);      // shared (reader) lock
}

// Atomics — lock-free where possible
std::atomic<int> counter{0};
counter.fetch_add(1, std::memory_order_relaxed);
++counter;                             // sequentially consistent
int expected = 0;
counter.compare_exchange_strong(expected, 1);

// Memory orders: relaxed < acquire/release < acq_rel < seq_cst (default)
// Default seq_cst unless you can prove you need otherwise.

// Condition variable
std::condition_variable cv;
bool ready = false;
{
    std::unique_lock lk(m);
    cv.wait(lk, []{ return ready; });   // unlocks while waiting; re-locks before predicate
}
{
    std::lock_guard lk(m);
    ready = true;
}
cv.notify_one();                        // or notify_all

// call_once — thread-safe initialization
std::once_flag init_flag;
std::call_once(init_flag, []{ /* runs exactly once */ });

#include <future>

// async — "run this and give me a future to its result"
auto fut = std::async(std::launch::async, []{ return compute(); });

// packaged_task — wrap a callable to produce a future
std::packaged_task<int(int)> task([](int x){ return x*x; });

Pitfalls, UB (Undefined Behavior), and surprises

  • Iterator invalidation. vector::push_back can reallocate, invalidating all iterators/references/pointers. map::insert never invalidates existing iterators (except for the affected node on erase).
  • Dangling references. auto& x = make_temp(); is fine only with const-ref-to-temporary lifetime extension. std::string_view sv = std::string("hi"); dangles immediately.
  • Object slicing. Storing derived in a base-by-value loses the derived parts and disables virtual dispatch.
  • Most vexing parse. Widget w(); declares a function, not an object. Use Widget w; or Widget w{};.
  • Initialization order across TUs. Order of construction of globals across translation units is undefined. Use the construct-on-first-use idiom (Meyers singleton) or constinit.
  • Signed/unsigned mixing. for (int i = 0; i < v.size(); ++i) compares signed to unsigned. Use std::ssize(v) or std::size_t.
  • operator[] on map creates entries. Use find or contains when you just want to check.
  • Auto-deducing the wrong type. auto x = matrix[i]; may copy a proxy. Prefer decltype(auto) or explicit types when proxies are involved.
  • Captured-by-reference lambda outlives its captures. Storing [&] lambdas in containers / handing them to other threads is dangerous.
  • Returning std::move of a local. Don't. Named Return Value Optimization (NRVO) elides the copy/move; std::move inhibits NRVO.
  • Two shared_ptr from one raw pointer. Double-free. Use make_shared or transfer through a unique_ptr.
  • Static initialization order across DLLs. Especially on Windows. Be very careful with globals in shared libraries.
  • Throwing from destructors. Calls std::terminate if it happens during another unwind.
  • Strict aliasing. Accessing an object through a pointer of a different (non-compatible) type is UB. Use std::bit_cast (C++20) or std::memcpy.
  • Data races on a single non-atomic. Even if you "think" the value is safely racing. Compiler may reorder reads/writes; the result is UB.
  • Self-move. x = std::move(x); is technically allowed (target left in valid-but-unspecified state) but almost always a bug.

Interview Q&A flash cards

What's the difference between std::vector and std::array?

vector is dynamically sized, heap-allocated, growable. array is fixed-size at compile time, lives on the stack (or wherever it's declared), with no allocator and no growth. array beats vector when the size is known and small/medium; vector when size varies or you need ownership semantics with move.

When should a destructor be virtual?

Whenever the class is intended to be a polymorphic base — i.e., if anyone might delete a derived object through a base pointer. Without it, the destructor call is non-virtual, only ~Base runs, and the derived destructor never executes → UB / leaks.

std::unique_ptr vs std::shared_ptr?

unique_ptr models exclusive ownership; it's move-only and has zero overhead vs a raw pointer. shared_ptr models shared ownership via an atomic reference count, with a control block allocated separately (unless you use make_shared, which fuses them). Prefer unique_ptr by default; reach for shared_ptr only when ownership truly is shared.

What's an rvalue reference and why do we need them?

T&& binds to rvalues (temporaries). They enable move semantics: a function can detect "this argument is about to die" and steal its resources instead of copying. They also enable perfect forwarding via "forwarding references" (also written T&&) in templates.

Explain RAII.

Resource Acquisition Is Initialization. Bind the lifetime of a resource (heap memory, file handle, lock, socket) to an object's lifetime: acquire in the constructor, release in the destructor. The stack unwinds correctly on exceptions, so cleanup is automatic. RAII is the foundation of exception-safe C++.

Why does std::vector::push_back sometimes invalidate iterators?

If the vector's size() == capacity(), push_back allocates a new buffer, moves/copies existing elements over, and frees the old buffer. All previous iterators, pointers, and references to the vector's elements are now dangling.

const T& vs T for function parameters?

Pass small/trivially-copyable types by value (int, std::string_view, std::span). Pass larger types by const T& to avoid the copy. The "pass-by-value-then-move" idiom is appropriate for sink parameters (constructors taking strings, etc.).

What's the difference between std::move and std::forward?

std::move unconditionally casts to an rvalue reference — you say "I'm done with this." std::forward<T> conditionally casts based on the deduced template parameter T, preserving whether the original argument was an lvalue or rvalue. Use std::forward only with forwarding references in templates.

Difference between map and unordered_map?

map is a balanced BST (typically red-black), keys are ordered, all operations are O(log n), and iteration is in sorted order. unordered_map is a hash table, O(1) average, O(n) worst, no ordering, needs a hash function. Use map when you need order or when keys are not easily hashable.

What is a virtual function table (vtable)?

The compiler implements virtual dispatch by giving each polymorphic class a hidden table of function pointers (the vtable). Each instance carries a hidden pointer (vptr) to its class's vtable. Calling p->foo() on a base pointer indirects through the vptr to call the correct override. Cost: one extra indirection per virtual call and one pointer per object.

Why is noexcept important on move operations?

STL containers (vector, etc.) use the strong exception guarantee. If a move can throw, the container falls back to copying during reallocation. Marking moves noexcept enables containers to move-relocate, which is much cheaper for non-trivial types.

What is the One Definition Rule (ODR)?

Every entity (function, variable, type) must have exactly one definition in the program, with limited exceptions (inline functions, templates, classes in different TUs — but the definitions must be token-identical). Violating ODR is UB and often produces nasty linker or runtime bugs.

What's the difference between constexpr and consteval?

constexpr functions may be evaluated at compile time if their arguments are compile-time constants; otherwise they run at runtime. consteval functions must be evaluated at compile time — runtime invocation is an error.

Why prefer emplace_back over push_back?

emplace_back forwards its arguments to the element's constructor, constructing the object in place inside the vector. push_back takes an already-constructed object and copies/moves it in. For types with non-trivial constructors this saves a move; for trivially copyable types the two are equivalent.

Differences between static_cast, dynamic_cast, reinterpret_cast, const_cast?

static_cast: compile-time, between related types (e.g., numeric, up/down a hierarchy without RTTI check) — unchecked. dynamic_cast: runtime, only for polymorphic types, returns null/throws on failure — uses RTTI. reinterpret_cast: bitwise re-interpretation (between unrelated pointer types, int ↔ pointer) — almost never portable. const_cast: add/remove const — modifying an originally-const object through it is UB.

Why is std::shared_ptr control block separate (without make_shared)?

When you write shared_ptr<T> sp(new T);, the T and the control block (refcount, weak count, deleter) are two separate allocations. make_shared<T>() fuses them into a single allocation, improving locality and saving an alloc — but the memory can't be freed until both strong and weak counts hit zero.

What's a "forwarding reference"?

A T&& where T is a template parameter being deduced. Reference collapsing rules make it bind to lvalues (yielding T&) or rvalues (yielding T&&). Combined with std::forward<T>(x), this preserves the value category of the original argument through the call.

What problem do concepts solve?

Template error messages and overload resolution. Before concepts, constraining a template required SFINAE tricks like std::enable_if, with cryptic errors when constraints failed. Concepts give you readable names (std::integral), readable errors ("constraint not satisfied"), and proper overload selection by specificity of constraints.

Why are coroutines useful?

They let you write asynchronous and lazy-sequence code in a linear, straight-line style — no explicit callbacks or continuations. The compiler transforms the function into a state machine that suspends at co_await / co_yield and resumes when its result is ready, while preserving local variables.

How does C++ handle multiple inheritance and the diamond problem?

By default, each base class is a distinct sub-object; with diamond inheritance D : B1, B2 where both inherit A, D contains two As. Use virtual inheritance (class B1 : virtual public A) to share a single A sub-object. Virtual inheritance has runtime/size costs and complicates constructors (most-derived class initializes the virtual base).

What is type erasure?

A technique for storing or operating on values of unrelated types behind a uniform interface, without virtual inheritance from a common base. Examples: std::function erases any callable with a matching signature; std::any erases any copyable type. Internally, type erasure typically uses an owned heap object plus a vtable-like dispatch table.

When should I use std::optional vs a pointer vs exceptions?

std::optional<T> for "may have a value" where ownership is by value and absence is expected (parse, lookup that may miss). Pointer / std::optional<T*> for non-owning references that may be absent. Exceptions for exceptional, unexpected, hard-to-handle locally errors — not for control flow.
